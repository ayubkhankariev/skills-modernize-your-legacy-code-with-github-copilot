# Test Plan — Student Accounts (COBOL)

Документ содержит тест-кейсы для проверки бизнес-логики текущего COBOL-приложения учётных записей студентов. Поля "Actual Result" и "Status" оставлены пустыми для заполнения после выполнения тестов с бизнес-стейкхолдерами.

| Test Case ID | Test Case Description | Pre-conditions | Test Steps | Expected Result | Actual Result | Status (Pass/Fail) | Comments |
|---|---|---:|---|---|---|---:|---|
| TC-001 | Уникальность идентификатора студента | Тестовый файл данных содержит запись с `student-id` "S100" | 1) Попытаться создать новую запись с `student-id` "S100" через CREATE-ACCOUNT. 2) Сохранить файл. | Система отклоняет создание дубликата; логирует ошибку дублирования; исходный файл не меняется. |  |  | Проверить логирование ошибки и поведение UI/ batch. |
| TC-002 | Создание аккаунта с обязательными полями | Нет записи с `student-id` "S200"; доступен файл данных | 1) Вызвать CREATE-ACCOUNT с полями: student-id, name, balance(0). 2) Записать в файл. | Запись присутствует в файле; все обязательные поля заполнены; операция залогирована как успешная. |  |  | Проверить корректность формата записи (FD). |
| TC-003 | Валидация обязательных полей | Подготовлен ввод с отсутствующим `name` | 1) Попытаться создать / обновить запись без `name`. 2) Записать. | Операция отклонена; запись не изменяется; сообщение в логе о валидации. |  |  | Проверить поведение для batch-режима (пропуск/стоп). |
| TC-004 | Применение частичного платежа | Существующая запись `S300` с balance = 1000 | 1) Выполнить APPLY-PAYMENT для `S300` на сумму 400. 2) Записать изменения. | Баланс уменьшается до 600; запись успешно сохранена; платёж залогирован. |  |  | Убедиться, что сумма корректно обновлена и формат чисел верный. |
| TC-005 | Применение полного платежа (обнуление) | Существующая запись `S310` с balance = 500 | 1) APPLY-PAYMENT на сумму 500. 2) Записать. | Баланс = 0; при бизнес-правиле аккаунт может быть помечен как "PAID"; запись сохранена. |  |  | Уточнить флаг статуса аккаунта после нуля. |
| TC-006 | Переплата (кредит на счёте) | Запись `S320` с balance = 200 | 1) APPLY-PAYMENT на сумму 300. 2) Записать. | Баланс становится -100 (переплата) или система оставляет 0 и отражает переплату как кредит; поведение должно соответствовать бизнес-правилу. |  |  | Зафиксировать требуемое поведение со стейкхолдерами. |
| TC-007 | Начисление пени/сборов | Запись `S330` с просроченной датой последнего платежа > N дней | 1) Выполнить CHARGE-FEE / CALCULATE-CHARGES цикл. 2) Записать изменения. | На баланс добавлена сумма пени; операция залогирована; расчёт соответствует формуле (фикс./% — уточнить). |  |  | Подготовить тестовые данные с известными датами для проверки формулы. |
| TC-008 | Чтение/запись повреждённой записи | Файл содержит строку с обрезанными полями/неправильным форматом | 1) Запустить обработку записей (PROCESS-RECORDS). | Повреждённая запись пропускается или помечается ошибкой; операция не ломает обработку других записей; запись в лог с подробностями. |  |  | Проверить режимы: fail-fast vs skip-and-log. |
| TC-009 | Отсутствие файла данных | Файл данных отсутствует (удален/пуст) | 1) Попробовать открыть файлы (OPEN). 2) Выполнить инициализацию / create new file. | Программа корректно создаёт новый файл (если предусмотрено) или сообщает об ошибке с понятным сообщением; не происходит краха процесса. |  |  | Проверить поведение в batch и interactive. |
| TC-010 | Закрытие аккаунта с непогашенной задолженностью | Запись `S340` с balance = 50 | 1) Попытаться пометить аккаунт `CLOSED`. 2) Сохранить. | Система запрещает закрыть аккаунт с положительным балансом; логирует причину; статус не меняется. |  |  | Уточнить политика для отрицательного баланса (переплаты). |
| TC-011 | Корректность транзакции чтение->запись | Набор записей для массового обновления | 1) Выполнить пакетную операцию: прочитать запись->изменить->записать для нескольких записей; симулировать сбой I/O на одной из записей. | После ошибки I/O: программа корректно логирует ошибку, не повреждает другие записи; может повторить/пропустить запись в зависимости от политики. |  |  | Смоделировать отказ файловой системы или права доступа. |
| TC-012 | Логирование ошибок и операций | Любая операция, приводящая к ошибке или успеху | 1) Выполнить операции: успешную запись, валидационную ошибку, I/O ошибку. | Для каждой операции — соответствующая запись в логе с временем, идентификатором записи, типом ошибки/успеха. |  |  | Проверить формат и местоположение логов.
| TC-013 | Производительность при большом объёме (скейлинг) | Файл данных с 10000+ записей | 1) Запустить batch-процесс обработки всех записей. 2) Измерить время и потребление ресурсов. | Процесс завершается за приемлемое время без потери данных; при длинной обработке логируются промежуточные статусы. |  |  | Определить "приемлемое" время со стейкхолдерами.
| TC-014 | Конкурентный доступ (если поддерживается) | Два процесса пытаются записать одну и ту же запись | 1) Процесс A читает запись. 2) Процесс B читает ту же запись. 3) A записывает изменение. 4) B пытается записать изменение. | Система должна обеспечить защиту от потери данных (блокировки/версионирование) или последняя запись отклоняется с уведомлением о конфликте. |  |  | В большинстве старых COBOL-сред блокировки управляет ОС/СУБД; зафиксировать ожидания.


## Инструкции по использованию тест-плана

1. Для каждой строки заполните колонку "Actual Result" по итогам выполнения.
2. В колонке "Status" укажите Pass/Fail.
3. Если требуется дополнительный тестовый набор данных — создайте файл `tests/fixtures/*.dat` и опишите его в комментариях в колонке "Comments".
4. После согласования с бизнес-стейкхолдерами можно транслировать эти кейсы в unit/integration тесты в node.js (например, mocha/jest + фикстуры).
